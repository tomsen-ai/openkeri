# Core Contract

openkeri is a runtime for building adaptive teacher agents.

## Current Product Track

The current active product work is **Learning Manager / Plan Studio** in
`examples/learning_manager/`.

The algorithm tutor described below is an early reference implementation for
the teacher-agent runtime. It remains useful background for schemas, memory,
evidence, and agent experiments, but it is not the current product focus.
The first productized learning track is a node-level operating-systems course
inside Plan Studio.

Plan Studio's active loop is:

```text
raw learning intent
-> dynamic intake
-> plan brief
-> editable learning mind map
-> node-level learning page
```

The active Plan Studio UI keeps graph-level concepts deliberately small:
`goal`, `stage`, `learn`, and `project`. Detailed explanation, practice, labs,
resources, review prompts, and acceptance criteria should live inside node
details instead of becoming default top-level graph node types.

It is not a general chatbot framework, and it is not a course generator. The
goal is to help a teacher agent observe a learner's current action, use learning
memory and evidence, produce a diagnosis, and choose an appropriate teaching
action.

## First Use Case

The first reference use case is a node-level operating-systems course for adult
technical learners.

A learner opens a course node, studies a lesson, works through practice, asks
questions, and records notes. openkeri helps the teacher agent diagnose the
learner's current issue and respond with either a hint or an explanation.

## Learning Loop Scope

openkeri should not become a single-step explanation engine.

Teaching is a learning loop:

```text
diagnose -> teach -> practice -> assess -> remediate -> review -> update memory
```

The first implementation only covers the smallest deterministic slice of that
loop:

```text
diagnose -> hint/explanation -> update memory
```

This is intentional. The v0 runtime proves the core loop with structured
diagnosis, teaching action, evidence, and memory. Later versions should extend
the runtime toward practice generation, assessment, mistake tracking, review,
and adaptive learning paths.

## Core Input

The teacher agent receives two input streams.

### 1. Current Input

Current input describes what is happening in the current learning turn.

It may include:

```json
{
  "problem": {
    "title": "Longest Substring Without Repeating Characters",
    "description": "Given a string s, return the length of the longest substring without repeating characters.",
    "target_concepts": ["sliding_window", "hash_map"]
  },
  "student_question": "I don't understand why my solution fails on abba.",
  "code_submission": {
    "language": "python",
    "code": "def lengthOfLongestSubstring(s):\n    ..."
  }
}
```

The current input answers:

- What problem is the learner working on?
- What is the learner asking?
- What has the learner submitted?

### 2. Memory Context

Memory context is generated by the Memory Module. It should not be raw chat
history.

It may include:

```json
{
  "session_state": {
    "session_id": "sess_001",
    "current_problem_id": "leetcode_3",
    "current_phase": "working",
    "recent_actions": ["submitted_code", "asked_about_abba"],
    "last_teaching_action": "hint",
    "hint_count": 1,
    "problem_status": "in_progress"
  },
  "learner_memory": {
    "learner_profile": {
      "learner_id": "learner_001",
      "level": "beginner_intermediate",
      "goal": "algorithm_interview_prep",
      "preferred_language": "zh",
      "teaching_style": "hint_first",
      "weak_concepts": ["sliding_window"],
      "common_mistakes": ["boundary_condition", "left_pointer_update"]
    },
    "related_history": [
      {
        "concept": "sliding_window",
        "issue": "left_pointer_update",
        "seen_count": 2
      }
    ]
  }
}
```

The memory context answers:

- What should the teacher remember about this learner?
- What has happened recently in this session?
- What long-term patterns are relevant to the current problem?

## Memory Module

The Memory Module manages learning memory.

It is responsible for:

- session state as short-term memory
- long-term learner memory
- learning event records
- learner preferences
- related history retrieval
- memory context generation
- memory updates after each teaching turn

The Memory Module should turn past learning records into useful teaching
context.

It should not blindly pass full chat history into the teacher agent.

## Evidence Layer

The Evidence Layer turns the learner's current action into facts that can
support diagnosis.

For the first course use case, evidence may include:

```json
{
  "items": [
    {
      "id": "ev_001",
      "type": "execution_result",
      "source": "python_code_runner",
      "summary": "The code fails on 1 of 4 test cases.",
      "content": {
        "status": "failed",
        "passed_count": 3,
        "failed_count": 1,
        "failed_cases": [
          {
            "input": "abba",
            "expected": 2,
            "actual": 3
          }
        ],
        "error": null
      }
    }
  ]
}
```

The Evidence Layer is not responsible for teaching.

It only provides facts, such as:

- code execution result
- failed test cases
- runtime errors
- basic code behavior

The teacher agent uses this evidence to produce a diagnosis.

## Core Output

The teacher agent outputs two main objects:

```text
diagnosis
teaching_action
```

### 1. Diagnosis

Diagnosis describes what the teacher agent believes is happening.

Example:

```json
{
  "status": "incorrect",
  "issue": "left_boundary_update_error",
  "concept": "sliding_window",
  "evidence": "The submitted code fails on input 'abba'."
}
```

Diagnosis answers:

- Is the learner's current attempt correct?
- What is the main issue?
- Which concept is involved?
- What evidence supports this diagnosis?

### 2. Teaching Action

Teaching action describes how the teacher agent responds.

The first version supports only two teaching action types:

```text
hint
explanation
```

A hint gives direction without revealing the full answer.

```json
{
  "type": "hint",
  "message": "Try tracing your code with input 'abba'. When the second 'b' appears, where should the left pointer move?",
  "next_expected_action": {
    "type": "revise_code",
    "instruction": "Update the left pointer logic and submit again."
  }
}
```

An explanation directly explains a concept or mistake.

```json
{
  "type": "explanation",
  "message": "In a sliding window, the left pointer should only move forward. When a duplicate character appears inside the current window, left should move to the position after the previous occurrence.",
  "next_expected_action": {
    "type": "revise_code",
    "instruction": "Apply this rule to your code and submit again."
  }
}
```

Teaching action answers:

- Should the teacher hint or explain?
- What should the learner see?
- What should the learner do next?

## Runtime Flow

The first version of openkeri follows this flow:

```text
current_input
+ memory_context
-> evidence_layer
-> diagnosis
-> teaching_action
-> memory_update
```

In more detail:

```text
1. Receive current input
2. Load memory context from the Memory Module
3. Collect evidence from the Evidence Layer
4. Produce a diagnosis
5. Choose a teaching action: hint or explanation
6. Return the teaching action to the learner
7. Store the learning event and update memory
```

## Non-Goals For The First Version

The first version does not aim to support:

- full course generation
- slide or PPT generation
- visual diagrams
- code annotation UI
- multi-agent classrooms
- gamified rewards
- K12-specific teaching flows
- a full LMS
- automatic full-solution generation by default

These may be added later, but the first version focuses on a small adaptive
teaching loop.

## Design Principle

The teacher agent should not simply answer.

It should:

```text
observe -> gather evidence -> diagnose -> choose a teaching action -> update memory
```

The core difference between openkeri and a chatbot is that openkeri treats
teaching as a structured runtime process, not a single response.
