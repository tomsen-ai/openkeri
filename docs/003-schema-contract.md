# Schema Contract

This document defines the first schema contract for openkeri.

The goal is to make the core data boundaries explicit before implementation.
These schemas are intended to be implemented with Pydantic v2 in the first
code version.

## 1. Schema Overview

The first version of openkeri uses these core schemas:

```text
LearningTask
CurrentInput
MemoryContext
Evidence
TeachingContext
Diagnosis
TeachingAction
TeacherOutput
LearningEvent
```

Their relationship:

```text
LearningTask
-> task-specific resources, such as Problem and test suites

CurrentInput
+ MemoryContext
+ Evidence
= TeachingContext

TeachingContext
-> TeacherAgent
-> TeacherOutput

TeacherOutput
= Diagnosis + TeachingAction

CurrentInput + Evidence + TeacherOutput
-> LearningEvent
```

## 2. CurrentInput

`CurrentInput` describes what is happening in the current learning turn.

It is provided by the learner or the caller. It is not memory, and it is not
evidence.

Example:

```json
{
  "problem": {
    "id": "leetcode_3",
    "title": "Longest Substring Without Repeating Characters",
    "description": "Given a string s, return the length of the longest substring without repeating characters.",
    "target_concepts": ["sliding_window", "hash_map"],
    "difficulty": "medium"
  },
  "student_question": "I don't understand why my solution fails on abba.",
  "code_submission": {
    "language": "python",
    "code": "def lengthOfLongestSubstring(s):\n    ..."
  }
}
```

### Problem

```text
id: str
title: str
description: str
target_concepts: list[str]
difficulty: str | None
```

`difficulty` is intentionally not an enum in the first version because different
problem sources may use different difficulty labels.

### CodeSubmission

```text
language: str
code: str
```

The first implementation may only execute Python, but the schema should not
hard-code that limit.

### CurrentInput

```text
problem: Problem
student_question: str | None = None
code_submission: CodeSubmission | None = None
```

`problem` is required because the teacher agent needs task context.

`student_question` and `code_submission` are optional so the same schema can
support:

- question only
- code submission only
- question plus code submission

## 3. LearningTask

`LearningTask` describes a reusable learning activity at a more general level
than an algorithm problem.

An algorithm problem is one task type, but future task types may represent a
reading task, writing task, rubric-based assessment, debugging task, or other
learning activity.

Example:

```json
{
  "id": "leetcode_3",
  "title": "Longest Substring Without Repeating Characters",
  "description": "Given a string s, return the length of the longest substring without repeating characters.",
  "task_type": "algorithm_problem",
  "target_concepts": ["sliding_window", "hash_map"],
  "difficulty": "medium",
  "metadata": {
    "source_problem_id": "leetcode_3"
  }
}
```

Schema:

```text
id: str
title: str
description: str
task_type: str
target_concepts: list[str] = []
difficulty: str | None = None
metadata: dict[str, Any] = {}
```

`LearningTask` is intentionally generic. Task-specific assets are not stored on
the schema directly. They are attached through a `LearningTaskBundle` resource
map by the task registry. For the current algorithm tutor, the bundle attaches a
`Problem` resource and a `ProblemTestSuite` resource.

## 4. MemoryContext

`MemoryContext` is generated by the Memory Module.

It is not raw chat history. It is a compact teaching context derived from
session state, long-term learner memory, and related history.

Example:

```json
{
  "session_state": {
    "session_id": "sess_001",
    "current_problem_id": "leetcode_3",
    "current_phase": "working",
    "recent_actions": ["submitted_code", "asked_question"],
    "last_teaching_action": "hint",
    "hint_count": 1,
    "problem_status": "in_progress"
  },
  "learner_memory": {
    "learner_profile": {
      "learner_id": "learner_001",
      "level": "beginner_intermediate",
      "goal": "algorithm_interview_prep",
      "preferred_language": "zh",
      "teaching_style": "hint_first",
      "known_concepts": ["array", "hash_map"],
      "weak_concepts": ["sliding_window"],
      "common_mistakes": ["boundary_condition", "left_pointer_update"]
    },
    "related_history": [
      {
        "concept": "sliding_window",
        "issue": "left_pointer_update",
        "seen_count": 2,
        "last_seen_problem_id": "leetcode_3"
      }
    ]
  }
}
```

### SessionState

```text
session_id: str
current_problem_id: str | None = None
current_phase: str = "working"
recent_actions: list[str] = []
last_teaching_action: str | None = None
hint_count: int = 0
problem_status: str = "in_progress"
```

`SessionState` is short-term memory for the current learning session. It records
where the current problem stands, but the Runtime decides how to advance the
flow.

`hint_count` helps the teacher agent decide whether to keep hinting or move to
an explanation.

### LearnerProfile

```text
learner_id: str
level: str | None = None
goal: str | None = None
preferred_language: str = "en"
teaching_style: str = "hint_first"
known_concepts: list[str] = []
weak_concepts: list[str] = []
common_mistakes: list[str] = []
```

The first version does not include mastery scores. This avoids adding
false precision before the update rules are clear.

### RelatedHistoryItem

```text
concept: str
issue: str
seen_count: int = 1
last_seen_problem_id: str | None = None
```

### LearnerMemory

```text
learner_profile: LearnerProfile
related_history: list[RelatedHistoryItem] = []
```

### MemoryContext

```text
session_state: SessionState
learner_memory: LearnerMemory
```

## 5. Evidence

`Evidence` is a set of facts collected from tools, execution, retrieval, or the
environment to support diagnosis.

Evidence is not user input. Evidence is also not analysis.

```text
CurrentInput = what the learner gave us
MemoryContext = what history tells us
Evidence = what external tools or environment observed
Diagnosis = what the teacher agent infers
```

For the first implementation, Evidence mainly contains code execution results.

Example:

```json
{
  "items": [
    {
      "id": "ev_001",
      "type": "execution_result",
      "source": "code_runner",
      "summary": "The code fails on input 'abba'.",
      "content": {
        "status": "failed",
        "passed_count": 8,
        "failed_count": 1,
        "failed_cases": [
          {
            "input": "abba",
            "expected": 2,
            "actual": 3
          }
        ],
        "error": null
      }
    }
  ]
}
```

### EvidenceItem

```text
id: str
type: str
source: str
summary: str | None = None
content: dict[str, Any]
```

The generic `EvidenceItem` shape keeps the schema extensible. Future evidence
types may include:

- execution results
- runtime errors
- complexity analysis
- static analysis
- retrieved documentation
- experiment results
- formula checks

### FailedCase

For execution evidence, failed cases use:

```text
input: Any
expected: Any
actual: Any
```

### ExecutionResult

For execution evidence, `content` may follow:

```text
status: "passed" | "failed" | "error" | "not_run"
passed_count: int = 0
failed_count: int = 0
failed_cases: list[FailedCase] = []
error: str | None = None
```

### Evidence

```text
items: list[EvidenceItem] = []
```

## 6. TeachingContext

`TeachingContext` is the final structured context passed to the Teacher Agent.

It is assembled by the runtime.

```text
TeachingContext = CurrentInput + MemoryContext + Evidence
```

Example:

```json
{
  "current_input": {},
  "memory_context": {},
  "evidence": {}
}
```

Schema:

```text
current_input: CurrentInput
memory_context: MemoryContext
evidence: Evidence
```

The Teacher Agent should base its diagnosis and teaching action on
`TeachingContext`, not on unstructured prompt text.

## 7. Diagnosis

`Diagnosis` is the teacher agent's structured judgment about the current
learning state.

It is not learner-facing feedback. It is used to choose a teaching action and
update memory.

Example:

```json
{
  "status": "incorrect",
  "issue": "left_boundary_update_error",
  "concept": "sliding_window",
  "confidence": 0.84,
  "evidence_refs": ["ev_001"],
  "evidence_summary": "The submitted code returns 3 for 'abba', but expected 2."
}
```

Schema:

```text
status: "correct" | "incorrect" | "unclear" | "no_submission"
issue: str | None = None
concept: str | None = None
confidence: float
evidence_refs: list[str] = []
evidence_summary: str | None = None
```

`confidence` should be constrained to the range `0.0 <= confidence <= 1.0`.

The first version keeps only one main `issue` and one main `concept` so each
teaching turn stays focused.

## 8. TeachingAction

`TeachingAction` describes how the teacher agent responds to the learner.

The first version supports only two action types:

```text
hint
explanation
```

Example:

```json
{
  "type": "hint",
  "message": "Try tracing your code with input 'abba'. When the second 'b' appears, where should the left pointer move?",
  "next_expected_action": {
    "type": "revise_code",
    "instruction": "Update the left pointer logic and submit again."
  }
}
```

### NextExpectedAction

```text
type: "revise_code" | "answer_question" | "ask_followup" | "continue"
instruction: str
```

### TeachingAction

```text
type: "hint" | "explanation"
message: str
next_expected_action: NextExpectedAction | None = None
```

## 9. TeacherOutput

`TeacherOutput` is the result produced by the Teacher Agent.

```text
TeacherOutput = Diagnosis + TeachingAction
```

Example:

```json
{
  "diagnosis": {
    "status": "incorrect",
    "issue": "left_boundary_update_error",
    "concept": "sliding_window",
    "confidence": 0.84,
    "evidence_refs": ["ev_001"],
    "evidence_summary": "The submitted code returns 3 for 'abba', but expected 2."
  },
  "teaching_action": {
    "type": "hint",
    "message": "Try tracing your code with input 'abba'. When the second 'b' appears, where should the left pointer move?",
    "next_expected_action": {
      "type": "revise_code",
      "instruction": "Update the left pointer logic and submit again."
    }
  }
}
```

Schema:

```text
diagnosis: Diagnosis
teaching_action: TeachingAction
```

## 10. LearningEvent

`LearningEvent` is a structured record of one teaching turn.

It is not the learner profile, and it is not the memory context. It is the raw
event record from which memory can later be updated or summarized.

Relationship:

```text
many LearningEvent records
-> Memory Module summarizes them
-> MemoryContext for future turns
```

Example:

```json
{
  "event_id": "evt_001",
  "learner_id": "learner_001",
  "session_id": "sess_001",
  "current_input": {},
  "evidence": {},
  "teacher_output": {},
  "timestamp": "2026-05-20T10:30:00Z"
}
```

Schema:

```text
event_id: str
learner_id: str
session_id: str
current_input: CurrentInput
evidence: Evidence
teacher_output: TeacherOutput
timestamp: datetime
```

The first version does not store the full `memory_context` in every
`LearningEvent`. If this becomes necessary later, a `memory_snapshot_id` can be
added.

## 11. Schema Principles

The first schema version follows these principles:

- Keep user input, memory, evidence, diagnosis, and action separate.
- Prefer structured objects over prompt-only conventions.
- Keep the first version small enough to test.
- Make evidence extensible through `EvidenceItem`.
- Keep each teaching turn focused on one main issue and one main concept.
- Avoid mastery scores until update rules are clear.
- Store raw learning turns as `LearningEvent` records.
