第 1 章 · 内核引导

从零写一个最小内核,让它开机打出 Hello

做完本章,你会有一个能在裸机(QEMU 模拟的 RISC-V)上启动、并在屏幕打出 Hello from my kernel! 的最小内核。更重要的是,你会真正理解:为什么这件"小事",其实要你亲手补齐三样东西。

一、你到底在写什么?

平时你写一个程序,比如 printf("Hello"),一运行屏幕就出字。这背后,是操作系统在替你默默干活:把你的程序装入内存、让 CPU 去执行它、还提供了 printf 这样的库函数。你的程序,一直站在操作系统的肩膀上。

而本章你要写的,就是这个操作系统本身——确切说,是它最核心的那部分,叫内核(kernel)。一旦你坐到这个位置,脚下那一层就没了:没人帮你启动、没人给你内存、也没有 printf。

所以,把目标缩到最小——开机打出一行字——这件事会逼你补齐操作系统平时白送你的三样东西:

① 启动:让你的代码能被 CPU 跑起来 · ② 内存:让函数有地方运行(栈) · ③ 输出:让字符能送上屏幕。下面逐个来。

二、启动:你的代码凭什么会被跑到?

你可能以为,开机就直接运行你的代码。其实不是。开机那一瞬间,CPU 什么都不懂,不认识"程序",也不认识"操作系统"。它只会做一件最机械的事:跳到一个固定地址,执行那里的指令。

在我们这台机器上,开机后有一段叫固件的小程序(OpenSBI,相当于 PC 上的 BIOS/UEFI)比你先醒。它先把最底层的硬件准备好,然后跳到一个约定好的地址 0x80200000,把控制权交给你。

所以你的任务很简单:保证你的代码,正好待在 0x80200000 这个地址。谁来决定代码摆在内存的哪里?是一个叫链接脚本(linker script)的小文件——它定义了可执行文件的内存布局。

ENTRY(boot)                 // 入口函数叫 boot
SECTIONS {
  . = 0x80200000;           // 从这个地址开始摆放
  .text   : { KEEP(*(.text.boot)); *(.text .text.*); }
  .rodata : { *(.rodata .rodata.*); }
  .data   : { *(.data .data.*); }
  .bss    : { *(.bss .bss.*); }
  . = . + 128 * 1024;          // 预留 128KB 给栈
  __stack_top = .;             // 记下栈顶地址,第二步要用
}

三、内存:函数为什么一跑就崩?

代码能被跑到了。你写个再普通不过的 C 函数想干点事——结果它直接崩了。语法明明没错,为什么?

原因藏在一个你平时根本不操心的东西里:栈(stack)。每个 C 函数运行时,都需要一块"草稿区"来放局部变量、保存返回地址("我执行完要回到哪")。每调用一层函数,就在栈上"压"进一个这样的栈帧(stack frame);函数返回,栈帧弹出。CPU 用一个寄存器 sp(stack pointer)记着当前栈顶。

所以现在 sp 指着一片乱七八糟的内存,函数一往栈上写东西,就把别处的数据覆盖了 → 崩。修法只有一步:在跑任何 C 代码之前,先把 sp 指到我们预留好的栈顶 __stack_top。这件事必须用一点点汇编,在最开头的 boot 里做:

// boot 是入口:开机后固件跳到这。它做两件事:支栈、跳进 C
__attribute__((section(".text.boot")))
__attribute__((naked))
void boot(void) {
  __asm__(
    "mv sp, %[stack_top]\n"   // ← 承重墙:把栈指针指到栈顶
    "j kernel_main\n"            // 然后跳进真正的主函数
    :: [stack_top] "r" (__stack_top));
}
🧱 承重墙mv sp, __stack_top 这一行是本步的核心:没有它,后面任何 C 函数都跑不起来。要真懂"为什么先支栈"。
⬛ 当黑盒naked 属性的作用:让编译器别在函数前后自动插入"用栈"的指令(因为此刻栈还没好)。只在这种入口处用,照抄即可。

四、输出:没有 printf,怎么把字送上屏幕?

代码稳稳跑起来了。你想打印 Hello,伸手找 printf……没有。printf 本来就是操作系统(或它带的库)提供的,现在脚下空了,它也没了。

你自己是没法直接点亮屏幕的——那是更底层、更高权限的事。但别忘了那个比你先醒的固件:它留了一个最小服务——"你给我一个字符,我替你显示"。你怎么叫它?用一条特殊指令 ecall:它让 CPU 从你的内核(S 模式)"陷入"到更底层的固件(M 模式),办完再切回来。

把这一声"喊"包成一个函数,就是你自己的、最小的 putchar;再用一个循环把整行字逐个送出去:

// 把一个字符交给固件显示(经 ecall);sbi_call 的汇编细节是黑盒
void putchar(char ch) {
  sbi_call(ch, 0, 0, 0, 0, 0, 0, 1);   // 1 号服务 = Console Putchar
}

void kernel_main(void) {
  const char *s = "Hello from my kernel!\n";
  for (int i = 0; s[i]; i++)
    putchar(s[i]);                  // ← 承重墙:一个个喊出去
}
⬛ 当黑盒sbi_call 内部怎么把参数塞进寄存器、怎么执行 ecall 指令——这套调用约定照抄即可。你要懂的是这一句:打印 = 把字符交给固件去显示。

你刚刚做的,是亲手补齐了操作系统平时白送你的三样:启动、内存、输出。这,就是"从零造一个操作系统"真正的含义——脚下什么都没有,每一样,你都自己来。
